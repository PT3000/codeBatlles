# 실시간 코드 배틀 플랫폼 상세 구현 문서

## 1. 프로젝트 개요

실시간 코드 배틀 플랫폼은 로그인한 사용자끼리 1:1로 매칭되어 동일한 알고리즘 문제를 풀고, 먼저 정답 판정을 받은 사용자가 승리하는 웹 서비스이다. 사용자는 자동 매칭 또는 특정 사용자 ID를 통한 배틀 신청으로 대결을 시작할 수 있으며, 배틀 중에는 Monaco Editor를 이용해 코드를 작성하고 상대방의 작성 상황을 실시간으로 확인할 수 있다.

본 프로젝트는 프론트엔드 `HTML`, `CSS`, `JavaScript`, 백엔드 `FastAPI`, 실행 환경 `Docker`, OS `Rocky Linux 9.7`, VM `VMware Workstation Pro`를 기반으로 구현한다. 개발 초기에는 FastAPI가 프론트엔드 정적 파일과 API를 함께 제공하고, 최종 배포 단계에서 필요 시 Apache를 추가한다. 코드 채점은 오픈소스 채점 서버 API인 `Judge0`를 활용하고, 코드 편집기는 `Monaco Editor`를 사용한다.

## 4. 핵심 기능 명세

## 4.1 회원가입

사용자는 아이디, 비밀번호, 닉네임을 입력하여 회원가입한다. 아이디와 닉네임은 중복될 수 없으며, 비밀번호는 해시 처리 후 DB에 저장한다.

### 입력 항목

| 항목 | 설명 | 필수 |
| --- | --- | --- |
| user_id | 로그인용 ID | O |
| password | 비밀번호 | O |
| nickname | 화면 표시 이름 | O |

### 처리 흐름

1. 사용자가 회원가입 정보를 입력한다.
2. 프론트엔드가 백엔드 회원가입 API로 데이터를 전송한다.
3. 백엔드는 아이디와 닉네임 중복 여부를 확인한다.
4. 문제가 없으면 비밀번호를 해시 처리하여 저장한다.
5. 기본 티어는 `브론즈`로 설정한다.

## 4.2 로그인 및 로그아웃

사용자는 아이디와 비밀번호로 로그인한다. 로그인 성공 시 JWT 토큰을 발급하고, 사용자의 온라인 상태를 활성화한다. 로그아웃 시 온라인 상태를 비활성화한다.

### 처리 흐름

1. 사용자가 로그인 정보를 입력한다.
2. 백엔드는 사용자 정보를 조회하고 비밀번호를 검증한다.
3. 검증 성공 시 JWT 토큰을 발급한다.
4. `users.is_online` 값을 `true`로 변경한다.
5. 로그아웃 시 `is_online` 값을 `false`로 변경한다.

## 4.4 자동 매칭

사용자는 자동 매칭 버튼을 눌러 현재 온라인 상태인 사용자 중 자신과 티어가 비슷한 사용자를 찾는다.

### 매칭 우선순위

1. 현재 로그인 중인 사용자
2. 배틀 중이 아닌 사용자
3. 본인과 티어가 비슷한 사용자
4. 매칭 대기 시간이 오래된 사용자

### 처리 흐름

1. 사용자가 자동 매칭을 요청한다.
2. 서버는 사용자를 매칭 대기열에 등록한다.
3. 서버는 온라인 상태이면서 배틀 중이 아닌 사용자를 검색한다.
4. 티어 차이가 가장 적은 사용자를 우선 매칭한다.
5. 매칭 성공 시 두 사용자에게 WebSocket으로 매칭 성공 이벤트를 보낸다.
6. 10초 카운트다운 후 배틀을 시작한다.

## 4.5 특정 사용자 배틀 신청

사용자는 상대방의 ID를 입력하여 배틀을 신청할 수 있다.

### 신청 조건

1. 상대방이 현재 로그인되어 있어야 한다.
2. 상대방이 이미 배틀 중이면 안 된다.
3. 상대방이 요청을 수락해야 한다.

### 처리 흐름

1. 요청자가 상대방 ID를 입력한다.
2. 서버는 상대방의 온라인 상태와 배틀 가능 상태를 확인한다.
3. 조건을 만족하면 상대방에게 배틀 신청 알림을 보낸다.
4. 상대방은 수락 또는 거절을 선택한다.
5. 수락 시 10초 카운트다운 후 배틀을 시작한다.
6. 거절 시 요청자에게 거절 알림을 보낸다.

## 4.6 배틀 시작

매칭이 성사되면 서버는 두 사용자에게 같은 문제를 배정하고 10초 뒤 배틀을 시작한다.

### 처리 흐름

1. 매칭 성공 이벤트 발생
2. 대기 화면에서 10초 카운트다운 표시
3. 서버가 랜덤 문제 또는 지정 난이도 문제를 선택
4. 두 사용자에게 동일한 문제 정보 전송
5. Monaco Editor 활성화
6. WebSocket 방을 생성하여 실시간 코드 동기화 시작

## 4.7 실시간 코드 공유

배틀 중 사용자가 코드를 작성하면 상대방 화면에도 해당 코드가 실시간으로 표시된다. 상대방의 코드는 읽기 전용 영역에 표시한다.

### 처리 방식

- 사용자의 코드 변경 이벤트를 JavaScript에서 감지한다.
- 변경된 코드 내용을 WebSocket으로 서버에 전송한다.
- 서버는 같은 배틀 방에 있는 상대방에게 코드 변경 내용을 전달한다.
- 상대방 화면의 읽기 전용 Monaco Editor에 코드가 반영된다.

### WebSocket 이벤트 예시

| 이벤트명 | 설명 |
| --- | --- |
| match_found | 매칭 성공 |
| countdown | 배틀 시작 전 카운트다운 |
| battle_start | 문제 공개 및 배틀 시작 |
| code_update | 코드 변경 내용 전달 |
| submit_result | 채점 결과 전달 |
| battle_end | 배틀 종료 |

## 4.8 코드 제출 및 채점

사용자가 코드 작성 후 제출하면 백엔드는 Judge0 API에 코드를 전달하고 채점 결과를 받아온다.

### 처리 흐름

1. 사용자가 코드 제출 버튼을 누른다.
2. 프론트엔드는 작성한 코드, 언어, 문제 ID를 백엔드로 전송한다.
3. 백엔드는 문제의 테스트케이스를 조회한다.
4. 백엔드는 Judge0 API에 소스코드와 테스트케이스를 전달한다.
5. Judge0의 채점 결과를 확인한다.
6. 모든 테스트케이스를 통과하면 정답 처리한다.
7. 정답자가 발생하면 서버는 즉시 배틀 종료 이벤트를 전송한다.

### 판정 결과

| 결과 | 설명 |
| --- | --- |
| Accepted | 정답 |
| Wrong Answer | 오답 |
| Time Limit Exceeded | 시간 초과 |
| Runtime Error | 실행 중 오류 |
| Compilation Error | 컴파일 오류 |

## 4.9 승패 처리

먼저 `Accepted` 판정을 받은 사용자가 승리한다. 승리자가 결정되면 배틀은 즉시 종료되며, 양쪽 사용자의 점수와 전적을 갱신한다.

### 처리 흐름

1. 정답 판정 발생
2. 승자와 패자 결정
3. 배틀 상태를 `finished`로 변경
4. 승자 점수 증가
5. 패자 점수 감소
6. 티어 재계산
7. 배틀 기록 저장
8. 결과 화면 표시

## 11. 개발 및 배포 서버 구성

개발 초기에는 Apache를 사용하지 않고 FastAPI가 프론트엔드 정적 파일, API, WebSocket을 함께 제공한다. 기능 구현이 완료된 후 최종 배포 단계에서 필요하면 Apache를 추가하여 정적 파일 제공과 `/api`, `/ws` 프록시 역할을 맡긴다.

### 개발 초기 요청 흐름

```text
사용자 브라우저
  -> FastAPI
    -> 정적 파일 요청: frontend 파일 응답
    -> /api 요청: FastAPI API 처리
    -> /ws 요청: FastAPI WebSocket 처리
```

### 최종 배포 요청 흐름

```text
사용자 브라우저
  -> Apache
    -> 정적 파일 요청: frontend 파일 응답
    -> /api 요청: FastAPI로 프록시
    -> /ws 요청: FastAPI WebSocket으로 프록시
```

## 12. Judge0 연동 방식

백엔드는 사용자의 제출 코드를 Judge0 API에 전달하고 결과를 받아온다.

### 제출 데이터

| 항목 | 설명 |
| --- | --- |
| source_code | 사용자가 작성한 코드 |
| language_id | Judge0 언어 ID |
| stdin | 테스트케이스 입력 |
| expected_output | 기대 출력 |
| cpu_time_limit | 시간 제한 |
| memory_limit | 메모리 제한 |

### 채점 처리

1. 문제의 테스트케이스 목록을 조회한다.
2. 각 테스트케이스마다 Judge0 제출 요청을 보낸다.
3. 모든 테스트케이스가 통과하면 `Accepted`로 처리한다.
4. 하나라도 실패하면 해당 실패 결과를 사용자에게 반환한다.

## 13. 보안 고려사항

- 비밀번호는 평문 저장 금지, 반드시 해시 저장
- JWT 토큰 기반 인증 적용
- 로그인 사용자만 매칭 및 배틀 참여 가능
- 본인이 속한 배틀방에만 WebSocket 접속 가능
- 관리자 API는 관리자 권한 사용자만 접근 가능
- 사용자 입력값 검증
- Judge0를 통한 코드 실행으로 백엔드 서버에서 직접 사용자 코드를 실행하지 않음

## 14. 예외 처리

| 상황 | 처리 |
| --- | --- |
| 매칭 중 사용자가 나감 | 매칭 대기열에서 제거 |
| 배틀 중 한 명이 접속 종료 | 일정 시간 재접속 대기 후 상대 승리 처리 |
| Judge0 응답 지연 | 제출 상태를 pending으로 표시 후 재조회 |
| 상대방이 배틀 요청 거절 | 요청자에게 거절 메시지 전송 |
| 상대방이 오프라인 | 배틀 신청 실패 메시지 표시 |
| 중복 제출 | 최근 제출을 기준으로 처리하되 배틀 종료 후 제출은 차단 |

## 17. 최종 시나리오

1. 사용자가 회원가입 후 로그인한다.
2. 메인 화면에서 자동 매칭 버튼을 누른다.
3. 서버가 온라인 사용자 중 티어가 비슷한 상대를 찾는다.
4. 매칭이 성사되면 10초 카운트다운이 시작된다.
5. 두 사용자에게 같은 문제가 공개된다.
6. 사용자는 Monaco Editor에서 코드를 작성한다.
7. 상대방의 코드 작성 상황이 실시간으로 표시된다.
8. 한 사용자가 코드를 제출한다.
9. 백엔드는 Judge0 API로 코드를 채점한다.
10. 정답이면 해당 사용자가 승리한다.
11. 서버는 승패, 점수, 티어를 갱신한다.
12. 결과 화면에서 승패와 점수 변화를 확인한다.
